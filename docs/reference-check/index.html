Reference-Check Demo

A hands-on, self-contained walkthrough of checking references and source-claims with agentic AI. You run two skills on a tiny project with deliberately planted errors, and practice the part no skill can do for you: deciding what to do about each flag.

Why this exists

A reference list is a promise: I read these, and they say what I claim they say. Two failures break that promise. The first is a citation to a source that does not exist. The second, quieter and far more common, is a citation to a real source that does not actually support the sentence it is attached to.

Large language models, used carelessly, manufacture the first kind at scale. Used carefully, the same technology is unusually good at catching both. This demo shows the careful version: not a chatbot you paste a question into, but structured skills that constrain what the model does, which tools it uses, and how it reports what it found.

The one rule. These tools are aids to judgment, not substitutes for it. Every flag is a prompt to go read the source yourself. Nothing here should be repeated in public, least of all the word fabricated, until a human has confirmed it.

1Set up an agentic environment

You need an AI tool that can read files in a folder and run reusable skills, not just a chat window. The reference setup is Claude Code, a command-line agent that works inside a project directory. Cursor, or any agent that supports the same skills format, works too.

Clone this demo so you have something concrete to run against:

git clone https://github.com/scdenney/ai-for-research.git
cd ai-for-research/demos/reference-check

2Install the skills

The checks live in the Open Science Skills plugin. Install it once, user-wide:

claude plugin marketplace add scdenney/open-science-skills
claude plugin install oss@open-science-skills

Restart the tool. Three skills carry this workflow:

research-repo · builds the knowledge base citation-check · audits the references fact-check · audits the claims

3Organize sources into a knowledge base

The claim check can only compare your sentences against sources it can read. That means the sources have to live on disk as machine-readable text, not as a pile of PDFs. The convention, scaffolded by research-repo, is one folder for the originals and one for the conversions:

sources/
├── og/   PDFs you keep locally (gitignored)
└── md/   one Markdown file per source — the knowledge base

In real work you drop a PDF in og/ and the intake step (process-source) converts it to md/<author>-<year>-<slug>.md. This demo ships the md/ files already written, so it runs offline. The rule that matters: the checker reads sources/md/, never the PDF.

4Run the reference check

Does each cited work exist, resolve, and parse? Does every in-text citation have a matching entry, and vice versa?

/oss:citation-check sample-project/manuscript.md sample-project/references.bib

On this project it should surface, among others:

WhatVerdict
othen2022 — "Digital Civics and the Participation Gap"Likely fabricated
vance2018 — cited in text, no entry in the bibliographyParity
kowalski2017 — in the bibliography, never citedParity
hainmueller2014 — a real methods paperVerified

That last row is the control. A checker that flags everything is as useless as one that flags nothing, so the demo keeps one real, correct reference to prove the difference. Full illustrative output: expected-output/citation-audit-report.md.

5Run the source-claim check

This is the deeper one. A reference can be real, well-formed, and resolve cleanly, and still be cited for something it never says. fact-check reads the source file behind each citation and asks whether it supports the specific claim.

/oss:fact-check sample-project/manuscript.md

It first runs a pre-flight gate: if the knowledge base is missing or does not cover the cited works, it refuses to run rather than reassure you falsely. Then, on this project:

ClaimSourceVerdict
"compulsory voting also raises citizens' political knowledge"ferreira-nair2021Contradicted
"higher social trust causes higher turnout"osei2020Overclaim
"modest, short-lived increase ... fades within two cycles"lindqvist2019Supported

The reference check passes ferreira-nair2021 and osei2020 without complaint. Only reading the source catches that one claim is backwards and another quietly upgrades a correlation into a cause. Formatting-clean is not the same as true. Full output: expected-output/fact-check-report.md.

6Interpret, then verify by hand

The report is a worklist, not a verdict. Do the human step on every flag:

1
Re-derive the evidence. Open the source file the report quotes. Read the passage. Does it say what the report says it says?
2
Confirm before you escalate. A "likely fabricated" flag can be a real paper the tool could not reach: paywalled, mis-keyed, grey literature. Fail to find it yourself before the word leaves your notes.
3
Decide the fix. "Causes" becomes "is associated with." That edit is yours to make, and yours to own.

The full accounting of every planted problem is in sample-project/ANSWER-KEY.md. Then try breaking the demo: add a real reference and watch it verify; fix a claim and watch Contradicted turn to Supported; delete a source file and watch the gate refuse to run.
